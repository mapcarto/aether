# Aether 服务体系架构 (Aether Service Architecture)

Aether 服务端架构是一套从底层到分布式、完全自主可控的实时时空计算平台。整体架构以**“极致性能、确定性延迟、解耦热更新”**为核心设计原则。依托 C 语言与 State Threads 协程模型，系统将计算核心与网络 I/O 严格剥离，基于进程边界与不可修改的**大页共享内存 (HugePages Shared Memory)**，构建了无锁且零拷贝的高级 IPC (进程间通信) 链路。

---

## 1. 核心组件矩阵 (Core Components)

整个服务端架构由四个高度解耦的技术模块构成：基础动态库、计算分片、路由控制面与无状态并发网关。

### 1.1 `libae.so` — 基础运算能力抽象层 (Atomic Core)
纯 C 实现底层动态库栈，封装全部硬核基础设施与数据层。作为计算算力的源泉，所有的上层应用（AP、AS）均强制链接 `libae.so`，共享同一套实现标准：
* **实体与组件 (ECS 框架)**：实体数据在底层内存中以 32 位内部索引的连续数组紧凑排列以榨取极高 L1/L2 缓存命中率。
* **金字塔时空结构引擎**：实现跨维度的 2D/3D/4D 空间切片，承担底层最高频次的 AOI 查询与碰撞干预逻辑。
* **时序与无锁环境生态**：内置时间轮 (Timing Wheel) 负责高效调度；提供大页**环形缓冲队列 (ringbuf)**、对象池、以及关键的原子锁操作。

### 1.2 Aether Pyramid (AP) — 单节点时空计算网格引擎
**角色定调**：单座“以太金字塔”引擎，负责封锁和计算一个专门的空间分片内的全部状态。
每个 AP 进程均是独立运行的计算单元，其核心逻辑由**看门人中枢 (Pyraman Orchestrator)** 驱动：
* **物理形态封装**：单实例包含一枚加载完整的 `libae.so` 以运行 ECS 和金字塔树，屏蔽不必要的 Socket 协议解析栈。
* **独立 IPC 缓冲区流水线**：为杜绝传统跨进程争抢，每个 AP 独占了互不打扰的内存交互管道：拥有一个专属入站请求用 `Request ringbuf`（由诸多 UDP 进程灌入，但读取端唯 AP / Pyraman 独尊），以及向对应 UDP 打回结果所需的 `Result ringbuf`。
* **事件信标驱动**：在内核空转时，依赖轻量的 `eventfd` 进行操作系统极速信号唤醒，避免空余周期拉升核心负载。

### 1.3 Aether Sphinx (AS) — 路由图谱与集群斯芬克斯
**角色定调**：“斯芬克斯”，守卫整个 AP 集群的领地拓扑入口控制面。\
**部署形态**：脱离具体业务与实体的轻量守护进程（支持同机或跨机的热备多部署），它是指路明灯：
* **分片动态映射账本**：巧妙复用了 `libae.so` 中的金字塔空间逻辑，但剥离了繁重的“实体”，纯粹仅维护一张全局“巨集空域图 → 所属 AP 进程列表” 的快速树结构。
* **边界网格拆单协调器**：接收 UDP 发出的低频超大尺度查询以及无头路由探针。对于刚好处于交界切割缝的查询请求，AS 将把需要涉入重算的复数关联 AP 标识打成集合打包返回给网关。负责监听 AP 心跳并下发宕机剔除策略。

### 1.4 ST UDP 层 — 并发排遣与协程防波堤
**角色定调**：完全无状态接入网关，阻挡和排解最残酷的直接网络峰涌。\
**部署形态**：极其粗暴有效的多进程横置（数量通常贴合机器真实 CPU 空余核数），彻底依靠内核网络底层自带的 `SO_REUSEPORT` 进行海量并发握手均摊。
* 通过内部极简化 C 的 State Threads (ST) 并发轮询：依靠底层 `st_poll` 同时监管双重外部入口——既监听外部庞杂的 UDP 网络数据包接收流，也一并实时兼顾那些来自内部 AP 计算引擎推发完毕后的 `eventfd` 就绪打唤。

## 2. 流式交互与数据管道 (Data Pipeline)

AP 引擎绝对不会去直面错配的外网连接端。这就要求有一套纯内侧极速无锁转运闭环作为支撑：

### 2.1 典型单分片聚焦管线 (Single-shard Fast Path)
标准地理区域请求的流转遵循极简的管线设计：
1. 客户端发出的 UDP 请求包进入某一个 ST UDP 网关进程完成初步解析。
2. UDP 根据坐标判定归属，若遇跨界或不明坐标，则向 AS 查询目的辖区的 AP 实例 ID（查询结果将在 UDP 侧进行短时缓存以提升下一帧命中率）。
3. 锁定目标 AP 后，UDP 直接将反序列化的请求载荷压入对应 AP 实例专用的 `Request Ringbuf`，同时调用 `eventfd_write` 唤醒引擎事件。
4. 等待挂起的 AP 进程被事件触发唤醒，在 `libae.so` 的支持下迅速完成实体位置更新及 AOI 范围探测，并将运算结果重新封装。
5. AP 通过溯源请求携带的标识，将结果数据投递至发送方 UDP 实例的 `Result Ringbuf` 中，随后触发 `eventfd` 通知该网关进程。
6. 被唤醒的 UDP 进程立刻读取结果队列，并经由标准网络层返回给末端客户端。

### 2.2 大范围扇出与聚合 (Cross-shard Fan-out / Reduce)
针对覆盖多个计算网格节点的大范围区域探测：
1. 客户端向 UDP 发送大范围空间或全图态势请求。
2. UDP 命中边界预警后向 AS 查询该区域包络，AS 返回覆盖多辖区的对应 AP 标识列表。
3. UDP 执行扇形派发机制（Fan-out），将相同的探测请求进行浅拷贝后，分别并行投入所有相关 AP 实例的独立请求队列中。
4. 各关联 AP 获取指令后进行互不干涉的并行计算处理，并在计算出的本地实体数据中打上“分片属地判定 (Shard Tag)”，随后投递回触发方 UDP。
5. UDP 进程在等待回调队列超时或收集完毕后进行归约操作（Reduce），将边界重叠区域产生冗余计算的实体进行滤重合并，最终合成完整全图态势回传客户端。

## 3. 分布式横向伸缩与安全热更演变

以简单和强制隔离对抗难以遏制的复杂度蔓延。

### 3.1 基于“多进程 + 绑核隔离”的确定性扩容
相较于在单体应用内处理复杂的 GC 调优或多线程锁竞争，Aether 选择依赖操作系统的应用实例多进程分发与资源物理隔离进行扩容应对。
* 针对因局部实体密度过高导致严重计算性能瓶颈的片区，无需在引擎底层堆砌复杂的降级自旋逻辑，而可直接在宏观管理台上将该分片网格的几何粒度继续截断拆解，在此新剥离的地块专属配发全新的空闲 AP 进程与已绑定的 CPU 处理核心予以负载分流。
* 作为无状态前端接入的 UDP 网关，天然支持通过横向增加守护进程与绑定空闲系统调度核心来应对外部极高连接潮汐并发数。

### 3.2 平滑无感的热更新交替 (Hot Relocations)
服务常态化在线要求严苛的无中断更迭演进能力。基于彻底隔离且仅共享内存边界的天然架构优势，Aether 摒弃了传统的全局停服重启发布：
当预部署的新版本 AP 实例完成冷启动加载并向核心 AS 寻址路由广播重新注册其所管理片域后，AS 会将后期所有针对该分片的请求路由全量更替至新 AP 节点。而在此时，处于老旧淘汰版本的残余旧版 AP 并不会遭遇系统强制中断杀退宣告（Kill）；它会被逻辑判定为被动孤立，但会照旧消费消化它内部请求环形内存里留存的残量事务与延迟回调包。一旦其本地持有队列完全扫光并长时间静默脱落系统级心跳后，该旧版老进程将会安全释放自回收退出（Graceful Exit）。整体换代过程极度绵密，前置接发无感，也并不引发应用链路对客户端有震动抛错感。

## 4. 部署红线与生态占位 (Deployment & Ecosystem)

Aether 的系统定位并非传统意义上的“业务管理平台”，而是填补了国内在**“微秒级实时时空计算”**领域的底层空白。因此必须严格遵守以下部署边界规范：

### 4.1 绝对的物理原生部署原则 (Bare-metal Native)
**禁止容器化妥协**：工业界（如常规航线审批或物流调度系统）主流方案往往采用云原生 (K8s) + PostgreSQL/PostGIS 架构。但那些关注的是“业务流”，而 Aether 处理的是高频致命的“时空流”，追求微秒级（而非百毫秒级）的动态避障响应。
由于系统高度依赖底层隔离进程间的共享大页内存 (HugePages) 以实现零拷贝通信，所有的实例必须紧贴物理机操作系统下沉部署（例如倾向具备 `SCHED_FIFO` 实时调度特性的国产操作系统如麒麟、欧拉）。如果轻易将 Aether 服务端打包进普通 Docker 虚拟化云容器栈，其跨界损耗将彻底击毁核心对于“延迟确定性”的承诺保障。

### 4.2 状态持久化与系统管辖边界
Aether 仅作为专业底层组件运作，不应盲目“变重”承揽冗余的行政任务：
* **存储哲学**：对于实体空间信息的暂存和落盘，坚持以“内存结构主宰 + Redis 轻度辅助持久化”为绝对方针。严禁在时空引擎热算力环内直连任何大本型关系数据库，以免引发不可逆转的查询 TPS 暴跌。
* **业务边界割裂**：所有的滞后性行政流程（如：任务派发审查、历史航迹回放、大屏监测）依然老老实实地放权回到基于 Kafka 等消息队列堆出来的全量企业级 Java Web 服务器去接管。外层业务只有在面临毫秒级“极密防撞测算”、“路径冲突硬判定”等需求时，才主动调用 Aether 切入的查询层网关来执行计算。

### 4.3 架构权衡与工程代价 (Engineering Trade-offs)
在构建微秒级的极低延迟与绝对时空确定性的同时，Aether 架构也摒弃了传统成建制的大数据管道与高并发中间件体系。作为决策前瞻，团队应当清晰直面如下架构落地所伴随产生的产品与工程成本：
1. **基层研发梯队建设要求严苛**：能够掌控重构底层原生 C 核心、协程机制 (State Threads)、Linux 透明大页与底层共享内存的原子屏障锁操作研发专员极其核心且人才储备较浅。针对这部分成员培养管理将不同于大量采用 Java/Go 作为主研语言体系的云计算全栈管理模式难度。
2. **剥离关系库原生容灾短板**：因拒绝引入基于 PostgreSQL 为代表且内嵌持久预写日志保障算法 (WAL-Log) 的重型网络延迟，本模型单纯挂载外围如 Redis 作为异步备份；当极端低概率出现大规模服务器物理主板突然断电或者 Kernel 定点崩溃，内存里未被导出备份成功的前数微秒高动态时空计算缓存状态极可能会有少许残余快照不能原样复现及绝对不丢防漏的承诺，存有轻微回弹重演风险。
3. **主流可观测面板与系统健康探针脱节隔离**：既然不能轻装上阵把系统容器化，Aether 自然也就没法享受诸如 Prometheus 原生接入打通或者 K8s 现成的 Pod 面板自动拓扑等红利。这就造成 Aether 所有的内部时移、健康拨测指标探测、环形共享堆溢出告警、或者 eventfd 的拥堵情况测查均须定制原生专向底层监控 Exporter 代理解析。这就无形变相大幅拉长了对可维护平台外延基础设施链搭建的工作负担。
