一句话定位：Aether 是面向高动态、高并发、强审计场景的时空计算底座，其核心价值在于“事件驱动并发 + 可回放可追溯 + 工程级集群编排”。

关键结论：在满足裸金属服务器部署（Bare-metal，即不使用虚拟化容器）、CPU 核心独占绑定（绑核）、大页内存启用（HugePages）、实时调度优先级等基础工程条件下，Aether 可作为城市级时空业务的统一底盘，向上兼容传统 GIS 客户端并向下承载 C 插件生态，形成"计算-渲染-AI"一体化链路。

1. 设计哲学与时空本体论 (Philosophy)

Aether 引擎为了解决在极大规模高动态并发场景下的“状态冲突、锁争用与不可追溯”难题，从底层架构上重塑了空间系统的世界观：

  • 万物皆事件 (Event-driven)：摒弃传统游戏引擎的“实体状态快照”修改方式。世界由瞬间构成、不断生灭的微小“事件”堆叠而成。依托 MPMC 无锁环形队列，实现系统事件并行投递与独立结算，彻底杜绝了并发过程中的多线程锁竞争。
  • 状态即过程与可溯源：基于底层内存池中的顺序事件记录，系统能够 100% 审计并回放出任意历史时刻的高精度物理切片，提供军工级的可审计能力。
  • Aether 时空本体论 (Ontology)：Aether 并非从零定义本体，而是将已在防务、卫生、制造等场景经过 Palantir 十余年验证的本体论范式引入高频空间计算，通过"构建-控制-反馈"三个环节形成全实时的时空本体闭环。语义层（构建阶段）将传感器数据映射为业务实体与拓扑关联，动力层（控制阶段）预定义符合物理约束的行动操作，动态层（反馈阶段）将现实变化转化为标准化事件流，维持毫秒级状态一致性。通过这一闭环，Aether 为 AI 代理介入现实业务流程提供了稳定的业务语义基础。

2. 快速部署与基础架构体系 (Quickstart)

Aether 在物理架构上明确分为两个层次：内核库级别的 Aether Kernel 和提供高并发网络支持的 Aether Server 集群。开发者可根据业务需求选择三种集成方式：

2.1 方式一：直联集成内核计算能力 (Aether Kernel)

适用于对延迟极其敏感的边缘计算场景，或需要将 Aether 空间计算能力深度集成到现有系统中的情况。业务方可直接链接纯 C 环境编写的底层动态库 libae.so。

  • 生命周期接管：开发者需要在业务代码中手动管理系统运算流程。包括调用 memarena_init 初始化大页内存 (HugePages)、创建支持 LOD 的实体金字塔索引，并通过主事件循环 (Main Loop) 驱动 Pyraman 调度器运行。
  • 注册事件钩子 (Event Hooks)：使用 dlopen/dlsym 或直接注册回调函数，将外部业务逻辑以事件形式挂载至无锁环形缓冲区 ringbuf。
  • 核心收益：以极低网络通信开销和纳秒级延迟，获得强大的并行时空计算能力。
# 引用头文件，将业务逻辑与 aether_core 动态库链接
gcc my_collision_plugin.c -laether_core -O3 -o my_standalone_engine

# 提升进程实时级别并启动，确保时钟确定性
chrt -f 99 ./my_standalone_engine

2.2 方式二：部署 Aether Server 工业集群网络

适用于省市级千万级并发推演的大型态势感知系统（如城市全域无人机统一调度）。此时开发者无需直接调用底层库，而是部署封装好的 Aether Server 微服务集群。

  • ST UDP Gateway (强并发接入层)：摒弃传统 Epoll 架构，采用基于 State Threads (ST 协程) 的高性能网关。配合 Linux 内核原生支持的 SO_REUSEPORT，将海量数据包均衡分发到多条处理流水线，单信道上下文切换延迟低于 200ns。
  • AP (Aether Pyramid 计算分片)：分布式算力矩阵的核心执行单元。每个 AP 进程绑定独立 CPU 核心，负责特定地理区域计算任务。通过本地缓存和底层 IPC 机制 eventfd 与网关联动，实现零内存拷贝的数据传递。
  • AS (Aether Sphinx 路由中枢)：负责协调多个 AP 分片，构建全局统一坐标网络。当查询请求跨越多个分片边界时，AS 执行 Map-Reduce 模式的聚合过滤，有效降低大规模并发查询复杂度。
部署红线限制：系统高度依赖底层硬件资源调度机制。为消除缺页中断、保障大页缓存效率以及精确控制 SCHED_FIFO 调度优先级与 NUMA 节点绑定，必须采用 Bare-metal (裸金属) 原生部署。严禁使用 K8s 或 Docker 等虚拟化容器平台，这些平台的高隔离特性会严重阻碍系统性能。

2.3 方式三：极速存量系统接入方案 (AE-EXT)

对于已拥有成熟三维可视化管线的业务方，Aether Server 提供了 AE-EXT (Aether Extension) 协议适配层，可实现向下兼容的无缝对接。

  • 显式寻址兼容：针对使用 3DTiles / I3S 架构的客户端，引擎能够实时监测空间体素变化，并以超低延迟动态生成 tileset.json 索引文件推送给前端。单次寻址平均延迟仅增加 0.8ms ~ 2.4ms，使前端系统感觉仍在请求普通静态文件。
  • 隐式网格回归：将 Aether 的高频事件推演结果伪装成传统数据格式。针对使用特定寻址规则的客户端（如 OSGB/PagedLOD 的文件目录树映射），AE-EXT 严格模拟目标系统的格网编号逻辑，让客户端完全按照既定习惯读取 Aether 的实时计算结果。

核心收益：开发者无需修改前端代码。只需将数据源 URL 指向 AE-EXT 节点，即可将传统基于静态离线数据的 GIS 服务，平滑升级为支持百万级目标实时推演的动态空间计算系统。单节点可稳定承载 5000+ 客户端的高频异步请求。

# 存量系统接入示例：仅修改数据源地址，前端代码零改动
# 原静态瓦片地址
url: "https://static-cdn.example.com/tiles/city/{z}/{x}/{y}.json"

# 替换为 AE-EXT 节点地址
url: "http://ae-ext.your-server:8080/tiles/city/{z}/{x}/{y}.json"
# AE-EXT 会实时从 Aether 引擎生成对应瓦片，接口保持完全兼容

3. 底层内核 (Core Subsystems)

Aether 的强大并发性能源自其内核中精简高效的 C 语言数据结构设计。这些组件共同构成了稳定可靠的核心运算引擎：

  • 动态体素金字塔网格 (Spatial Grid)：一套支持多层级细节 (LOD) 的离散网格系统，提供 O(1) 复杂度的模型查询能力。摒弃传统四叉树/八叉树方案中耗时的指针跳转与内存碎片问题，通过精巧的层次回调验证机制，快速完成碰撞检测与空间过滤。在大跨度对象场景下，系统采用“主实体 + 局部网格”的表达方式：主实体描述对象整体，局部网格仅表示该主实体在单个叶子网格上的占据登记，不表示一组子网格。
  • 无原型实体组件体系 (Archetype-less ECS)：为最大化 CPU 缓存命中率 (L1/L2)，系统采用扁平化内存布局策略，避免原型迁移时的数据拷贝开销。所有对象组件均以位掩码方式组织，实现 O(1) 时间复杂度的组件挂载与卸载。
  • Pyraman 看门人中枢与事件计时器：Pyraman（引擎内置的事件驱动调度器）将所有状态变更通过事件总线提交至 MPMC 无锁环形缓冲区。系统基于多层嵌套的时间轮结构调度回调请求，有效避免传统并发编程中的锁竞争 (Lock Contention) 问题。
  • 双金字塔原子场 (Double Pyramid)：专为处理气象云图、传感器探测等高速动态场数据而设计。通过在 Standby(备用) 和 Active(活动) 两个计算面之间进行“读写分离”，仅需一次原子指针交换操作 (Pointer Swap)，即可在微秒级时间内完成整个计算面的无缝切换。

3.1 事件数据生命周期

从工程治理角度，Aether 中每个事件都应被视为“可审计资产”。典型生命周期如下：

  1. 接入：外部消息由网关或本地插件写入 ringbuf，附带时间戳、实体 ID、空间索引与来源标记。
  2. 预筛：金字塔索引先执行 O(1) 候选过滤，剔除绝大多数无效对象组合。
  3. 结算：调度器按时间轮策略消费事件，并调用业务插件完成规则判定与动作生成。
  4. 发布：动作被封装为 action_event 推送总线，供渲染、告警或外部系统订阅。
  5. 归档：关键事件及快照被落地用于回放、审计、演练复盘与故障追踪。

4. API 扩展与集成契约规范 (API Reference)

Aether 将引擎底盘收编为绝对中立状态。这意味着所有关于行业专有审批、特定轨迹排拒规范算法逻辑必须以业务端扩展插件（Plugins）形式交付。它立足于三道工程纪律基线：

4.1 C ABI 插件沙箱机制

  • Bring Your Own Data/Algorithm (自带结构数据)：用户不需要推翻重写多年优化的底层算法，也不需要强行继承引擎对象基类。引擎只认 64 位 ID 和包围盒。用户专有复杂图谱数据可安稳放在自身内存里，引擎绝不越权强行解析。
  • 免费获得亿级并发加速：空间预筛是极耗算力的操作。用户只需编写单线程判断插件，底层金字塔结合 MPMC 会自动在后台将亿级对象筛选至最终数个候选实体再交给插件处理，黑盒算法也能直接享受并发加速。
  • 彻底避免 C++ 符号污染：通过 dlopen() 在冷启动或热更时挂载 .so / .dylib 模块，强制所有接口置于 extern "C" 下，从根本上消除 C++ 命名修饰带来的链接兼容问题。

4.2 内存安全边界

核心规则只有一条：引擎数据只读，插件不得持有任何引用。

  • 只读数据传递：引擎通过 const 常量指针向插件传递场景快照，插件只能读取数据，无法修改引擎内存。
  • 内存隔离要求：插件禁止向引擎传入 free()，也禁止将引擎地址存入静态变量。插件如需分配内存，须使用自己的 malloc 并自行负责释放。
/**
 * @brief 空间体素状态更新事件通知回调函数。
 * @param event 包含发生坐标及触发因子的不可变事件数据状态。
 * @param user_data 开发者注入的自定义业务上下文地址。
 * @warning 该回调将在无锁队列的消费线程中中异步拉起。严禁阻塞式系统调用（如原生阻塞 I/O）。
 */
typedef void (*voxel_update_cb_t)(const ae_event_t *event, void *user_data);

4.3 动作指令的安全下发机制

插件完成算法后，不能直接调用引擎接口下发动作。所有意图必须封装为 action_event_t，经由 ae_subscribe_event 送入无锁消息队列异步消费。即便插件发生死锁，独立于 Pyraman 看门人中枢运行的调度保护器（Orchestrator Guard）也能及时介入，保护整个服务集群不受影响。

5. 服务集群与全栈工具链生态 (Service & Toolchains)

在纯计算内核基础上，Aether 构建了完整生态系统，包括三维可视化监控和智能 AI 大语言模型代理网络：

5.1 Aether Server 分布式实时协同集群 (AE-Server)

作为封装内核并向外提供算力的分布式框架，AE-Server 基于 C 语言与协程模型，实现高并发 I/O 隔离与跨进程通信调度：

  1. 流式交互与跨分片编排 (Data Pipeline)：针对单一目标点查询请求，ST UDP 网关直接路由到对应 AP 的独立 Ringbuf，通过 eventfd 机制将轮询等待成本降至最低；面对覆盖超广域的聚合查询请求，利用 AS 执行全局 Fan-out 查询，将请求分发至数十个计算进程进行局部处理后再 Reduce 聚合，最终快速生成全局态势图。
  2. 物理级内存隔离与扩容 (Bare-metal Scaling)：拒绝使用复杂云化动态负载均衡算法。架构依赖底层大页表 (HugePages) 构建进程间 IPC 通道，通过强制 CPU 绑核机制实现资源切片和线性扩容。
  3. 零中断平滑热更新 (Graceful Updates)：通过共享内存实现业务状态分离。运维节点启动新版本进程并完成路由接管后，旧版本进程继续处理完队列中剩余任务后自动退出，实现计算网格层毫秒级无损版本迭代。
  4. 分布式一致性与故障恢复：AS 路由中枢通过原子快照与事件时间戳机制确保跨分片数据一致性（最终一致性模型）。网络分片场景下，系统采用“分片优先”策略，通过事件日志实现事后对账。单个 AP 故障时其他分片维持独立工作；故障 AP 重启后通过重放事件日志恢复至一致状态。
  5. 关键监控指标：环形队列深度（ms 延迟阈值 > 100 触发告警）、AP 分片心跳（< 2s）、跨分片 P95 延迟（目标 < 50ms）、事件丢弃率（< 10⁻⁶）。告警时调度保护器（Orchestrator Guard）自动触发故障转移，RTO < 5s。

5.2 AVA 视觉架构引擎枢纽 (Aether Visualization Architecture)

AVA 不是单纯图形库，而是连接内核与渲染后端的“智能翻译层”。它作为挂接在内核底层的“绝对单向只读观测层”，负责从高频事件流中提取数据并决定展示内容，但绝不反向干扰引擎时序运行。

  • Cairo 2D 渲染模块：无需遍历组件树，直接从线性内存数组中提取线段、热点坐标，由 Cairo 快速绘制。适用于构建大型战术指挥板与二维沙盘面板。
  • VKDK 管线架构 (Vulkan Dev Kit)：充分践行面向数据 (Data-Oriented) 设计原则。利用纯连续数组直接无缝对接 Vulkan 原生 SSBO (Shader Storage Buffer Object)，并通过 vkCmdDrawIndirect 间接绘制指令实现爆发式渲染。
  • 3DGS 动态高斯泼溅 (Gaussian Splatting)：面对数亿微粒堆叠的超大体量点云渲染，引擎通过金字塔建立 O(1) 复杂度的多级分辨率 LOD 索引。依靠只读快照机制提取渲染指令送至显卡进行分级过滤，支持无限细节的真实世界模型场景渲染。

5.3 大语言模型接入框架 (LLM Agent Framework)

针对大语言模型高度依赖 Context 窗口且"计算能力相对较弱"的特点，Aether 设计了防宕机事件中继架构，通过 MCP（Model Context Protocol，模型上下文协议） 标准化 AI 与引擎的交互边界，避免直接向 LLM 投喂原始坐标数据。在这个架构下，AI 与引擎交互被严格规范为六个阶段：

  1. 全局概览优先：初始接入时，不向模型传入原始坐标数据，只提供从金字塔 Level 0–2 层提取的高层概览信息，让 AI 先建立全局认知。
  2. 聚焦点拉取指令 (MCP Tools)：限制大模型只能通过调用预定义的原子技能 (MCP Tools)，才能获取指定聚焦区域内的细粒度矢量结构数据。
  3. 几何计算卸载：当 AI 检测到两个目标相互逼近时，绝对禁止 AI 自行计算几何布尔碰撞。所有复杂空间几何裁切与布尔运算必须通过专用中继打回 Aether 底座，由底层 C 算法库完成极值计算后仅返回 True/False 的确定性结论。
  4. 增量事件流订阅：在平稳监控阶段，LLM 不再执行高消耗的全量轮询 (Polling)，改为订阅 ringbuf 底层推送的增量事件流（如 ENTITY_MOVE），大幅降低数据传输负载。
  5. 可视化快照烘焙：LLM 在上报决策结果时，可调用下游 AVA 模块截取当前推演热区的 2D/3D 图像 (Base64 JPEG 格式) 作为辅助汇报依据。
  6. 数据令牌传递 (Token View)：当查询涉及海量体素时，框架禁止将数据序列化为文本传输。仅向大模型返回轻量的 VIEW_TOKEN_8F9A2C 令牌，由 LLM 将其转发给渲染客户端完成可视化，从根本上规避传输过载。

5.4 空间分析能力分级 (Spatial Analysis Domains)

通过逻辑解耦，Aether 建立了三个递进式空间分析计算管线：

  • 第一级格网域 (Grid Computing)：基于 Hilbert 曲线与空间降维编码，原生兼容 S2、H3 以及国家级三维正方体空间剖分标准。集成完整地图代数 (Map Algebra) 框架，支持高通滤波区域算子，并通过极速自适应反距离权重 (IDW) 算法及克里金 (Kriging) 最优无偏估计插值，还原大规模空间曲面场分布。
  • 第二级经典拓扑域 (Classical Computing)：依托 Martinez-Rueda-Feito (MRF) 布尔裁剪算法和 Point-in-Polygon (PIP) 环绕数验证法，执行高精度地理图形的剖切、拼接、合并、打洞等经典几何运算，确保拓扑关系绝对无损。
  • 第三级知识图谱域 (Knowledge SKG)：面向多要素关联分析。采用 DBSCAN/HDBSCAN 密度聚类算法识别空间热点区域，引入 Spatial PageRank 链路分析技术深入解析大规模动态交通网络、物流线路的传导关系与故障溯源。

6. 系统基准性能与节点容量预测分析 (Benchmarks & Capacities)

以下是 Aether 在大规模场景下的性能基准与容量上限：

6.1 核心并发性能指标

  • 索引查询吞吐量：在标准 x86 平台上（单 CPU 核心），依托 O(1) 索引结构与 memarena 无锁通道，系统可稳定维持超过 800 万 ~ 千万次/秒的事件查询与状态更新处理。
  • 高频动态实体容量：在高时延敏感、小尺度密集碰撞检测场景中，底层系统仍能支持20 万至 50 万+个高频动态对象持续进行状态更新，维持全场景 100Hz 刷新率稳定运行而不衰减。

6.2 城域级网格内存容量规划

部署层级说明：

  • Tier 1: 边缘微节点 (16G - 32G)：适用于基础设施受限的前线边缘设备、小规模空管与防碰撞现场控制终端。可管理近 170 亿个网格单元，并在核心区域维持 10 厘米级精确控制精度。
  • Tier 2: 重点分流服务器 (64G - 128G)：适配大容量高频运转服务器集群。足以支撑辐射范围超过几十公里的高频区域网络枢纽，提供从全局概览到局部 5cm 精度的高质量数据中转与分发能力。
  • Tier 3: 全国高性能算力群/行业中心云平台 (256G - TB 级)：全景容量支持容错性极高的全区域联动网络集群，充当应对数百万连接并发的视图渲染与计算主脑。

6.3 快速容量估算方法（工程实践）

在项目早期评审，可用以下简化方法估算资源需求：

目标体素总量 V = 覆盖体积 / 单体素体积
内存基线 M_base = V x 每体素字节数
并发余量 M_total = M_base x (1 + 事件峰值系数 + 回放保留系数)
CPU 核数建议 C = (峰值事件吞吐 / 单核稳定吞吐) x 安全系数

建议工程上采用“先保守、后压测、再扩容”策略：先按 P95 负载做容量规划，待接入真实业务流量后，依据监控指标回填模型并滚动修正参数。

6.4 扩容顶线与性能衰减分析

AP 分片管理与 AS 聚合瓶颈：

  • 单 AS 节点管理 16-32 个 AP 分片时，跨分片查询延迟 P95 < 100ms；超过 64 个分片时衰减至 > 300ms。
  • 单集群配置（4 核 CPU）建议 AP 分片总数不超过 8 个，超过此数时 AS 聚合环节开销呈线性增长。
  • 极限可扩展至 128 个分片，但会伴随明显性能衰减；此时建议采用多 AS 协调或升阶配置突破单点瓶颈。

7. 实施路线与工程治理建议 (Implementation Roadmap)

为了降低上线风险，推荐将 Aether 的落地拆分为三个阶段：

7.1 Phase A：技术验证 (PoC)

  • 选取 1 至 2 个高价值业务流程，定义明确成功指标（时延、吞吐、误报率、回放一致性）。
  • 构建最小可运行链路：网关接入、核心事件、单插件判定、可视化回传。
  • 完成基线压测，输出瓶颈定位报告与硬件规格建议。

7.2 Phase B：灰度并行

  • 采用双写或镜像流量策略，让旧系统与 Aether 并行运行，持续对比结果一致性。
  • 建立异常处置机制：队列积压、热点分片失衡、插件超时、事件风暴等故障预案。
  • 通过可观测性看板建立“事件速率-处理时延-资源占用”三轴监控闭环。

7.3 Phase C：生产切换

  • 按区域或业务域分批切换，优先从低风险场景开始，逐步扩大覆盖范围。
  • 启用热更新机制，确保插件版本迭代与回滚流程可验证、可演练、可审计。
  • 建立季度容量复盘制度，持续修正分片策略、绑核策略与存储保留策略。
治理建议：将“性能压测、灾备演练、审计回放一致性验证”纳入每次版本发布门禁，而非上线后的补救动作。
