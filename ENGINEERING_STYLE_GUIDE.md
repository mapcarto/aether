# Aether 工程技术文档编写规范 (Engineering Style Guide)

## 1. 核心准则 (Core Principles)
*   **平实 (Plain)**：禁止使用文学比喻、拟人、夸张修辞。
*   **严谨 (Rigorous)**：使用中性的工程术语，逻辑链条必须闭环。
*   **去情绪化 (De-emotionalized)**：文档是给工程师看的，禁止出现“痛点”、“魔咒”、“王炸”、“小钢炮”等情感驱动或口语化描述。
*   **去 AI 性 (Anti-LLM-Flavor)**：避免“正如...”、“总之...”、“令人兴奋的是...”这类典型的 LLM 润色话术。

## 2. 禁词表与替代建议 (Restricted Terms & Alternatives)
| 禁用内容 (Forbidden) | 推荐替代 (Recommended) |
| :--- | :--- |
| 打破...魔咒 / 痛点 | 解决...工程限制 / 业务背景 |
| 小钢炮 / 暴力性能 | 高性能边缘节点 / 高吞吐率 |
| 兵书 / 葵花宝典 | 技术方案 / 全量文档 |
| 战斗力表现 | 性能评估 / 指标属性 |
| 优雅 / 丝滑 | 低延迟 / 高兼容性 / 逻辑闭环 |
| 拥抱 AI / 认知解读端 | AI 模型集成接口 / 认知处理逻辑 |

## 3. 结构要求 (Structure)
*   所有文档必须以**事实 (Fact)** 和 **数据 (Data)** 驱动。
*   每一个结论必须对应一个具体的技术实现细节。
